# طراحیِ خطِ موازیِ نفس‌ها (Throughput-Only) — ۲۰۲۶-۰۷-۲۲

> مأموریتِ باغبان: فقط توانِ عملیاتی؛ نه الگوریتمِ کشف عوض شود، نه
> اعتبارسنجی‌ها، و خروجیِ علمی باید **بایت‌به‌بایت** با اجرای ترتیبی یکی
> باشد. اصل: ساده‌ترین راهی که چند عامل را هم‌زمان به کار بگیرد.

## ۰. واقعیتِ ساختاری که طراحی را شکل می‌دهد

استعارهٔ ساختمان یک اصلاح می‌خواهد: نفس‌ها آجرهای یک دیوارِ آزادِ پهن
نیستند؛ **رج‌های یک دیوارند** — هر رج بر رجِ پیشین می‌نشیند:

1. **گزینشِ ریشهٔ نفسِ N+1 به خروجیِ نفسِ N وابسته است** — شکاف‌های قویِ
   تازهٔ نفسِ N واردِ صف می‌شوند و قاعدهٔ «نادرتر مقدم» می‌تواند همان‌ها
   را جلو بیندازد.
2. **بایتِ رکوردِ نفسِ N تابعِ تاریخِ زیستهٔ ۱..N−1 است** (فرافکنی بر
   فهرستِ زیسته، `breath_no`، رویدادهای صف).

پس «چند Worker که هر یک نفسِ مستقلِ خودش را بردارد» با قیدِ «خروجیِ
یکسان با اجرای ترتیبی» **منطقاً ناسازگار** است. اما خبرِ خوب: حلقهٔ
وابستهٔ زنجیره (محاسبهٔ رکورد + به‌روزرسانیِ صف) **قطعی و چندثانیه‌ای**
است؛ بخش‌های کُند (تفسیر/روایت، سیمای تدبّر، ممیزی‌ها، آزمون‌ها) به
محتوای رکوردِ آماده وابسته‌اند ولی **به هم وابسته نیستند**. راه‌حلِ ساده:
**هستهٔ سریالِ نازک + خطِ مونتاژِ موازی**.

## ۱. معماری — سه ایستگاه

```
                    ┌─────────────────────────────────────────────┐
                    │  S0: برنامه‌ریز (سریال، ~ثانیه‌ای بر نفس)    │
                    │  زنجیرهٔ قطعی: رکوردِ N ← صفِ به‌روز ← ریشهٔ  │
                    │  N+1 ← رکوردِ N+1 ← … (K گام جلوتر)          │
                    │  خروجی: job با بایتِ رکورد + checksum        │
                    └────────────┬────────────────────────────────┘
                                 ▼  صفِ کار (jobs.json)
        ┌────────────┬────────────┬────────────┬────────────┐
        │ Worker ۱   │ Worker ۲   │ Worker ۳   │ Worker W   │   S1: غنی‌سازی
        │ نفس N      │ نفس N+1    │ نفس N+2    │ نفس N+k    │   (موازی، کُند)
        │ tadabbor · observe · audit-slice · پیش‌نویسِ دفتر │
        │ فقط در pipeline/work/<N>/ می‌نویسد                │
        └────────────┴──────┬─────┴────────────┴────────────┘
                            ▼  به ترتیبِ شمارهٔ نفس
                    ┌─────────────────────────────────────────────┐
                    │  S2: ثبت‌کننده (سریال، تک‌نویسنده)           │
                    │  دروازهٔ هم‌ارزی: بازتولیدِ رکورد از حالتِ    │
                    │  زنده == بایتِ برنامه‌ریزی‌شده؟ سپس: record + │
                    │  seed + seed-db + دفتر + ponder-sync +       │
                    │  pytest + verify + commit                    │
                    └─────────────────────────────────────────────┘
```

- **S0 برنامه‌ریز:** با همان `engine.breathe_record` (پارامترهای صریحِ
  queue/lived/breath_no) زنجیره را K گام جلو می‌بَرد — بدونِ دست‌زدن به
  life.db. چون همه‌چیز قطعی است، «شبیه‌سازی» همان «اجرا»ست؛ ریشهٔ هر نفسِ
  آینده از پیش و به‌درستی معلوم می‌شود. تخصیصِ خودکار = همین (انتظارِ ۱).
- **S1 کارگرها:** هر Worker یک job برمی‌دارد (قفلِ فایلِ اتمی؛ انتظارِ ۲)
  و فقط کارهای مستقل‌از‌هم را می‌کند: سیمای تدبّر، مشورتِ رصدخانه، برشِ
  ممیزیِ v2 برای مرکزِ تازه، پیش‌نویسِ روایتِ دفتر از روی رکورد. هیچ
  Worker به مخزنِ اصلی نمی‌نویسد — فقط به پوشهٔ کاریِ خودش.
- **S2 ثبت‌کننده:** تک‌نویسندهٔ مخزن، به ترتیبِ شمارهٔ نفس. هیچ merge
  واقعی لازم نیست چون فقط او می‌نویسد (انتظارِ ۶). پیش از هر ثبت،
  **دروازهٔ هم‌ارزی**: رکورد را از حالتِ زندهٔ واقعی بازتولید می‌کند و با
  بایتِ S0 مقایسه (checksum)؛ نابرابری ⇒ کلِ برنامهٔ باقی‌مانده باطل و
  بازبرنامه‌ریزی (epoch جدید). این قید، «یکسانی با اجرای ترتیبی» را از
  فرض به **ناوردای سنجیده** تبدیل می‌کند (انتظارهای ۷ و قیدِ طلایی).

## ۲. صفِ کار و فایل‌ها

```
pipeline/
  state.json      { epoch, last_committed_breath, planned_upto }
  jobs.json       [ { breath_no, root, epoch, record_sha256,
                      status: pending|running|completed|failed|reviewed,
                      worker, attempts, timestamps } ]
  work/<no>_<root>/   record.json (از S0) · portrait.md · observe.json ·
                      audit_slice.json · ledger_draft.md · done.flag
  locks/<no>.lock     قفلِ اتمیِ برداشتنِ job (O_CREAT|O_EXCL — stdlib)
```

- وضعیت‌ها (انتظارِ ۴): `pending` (S0 ساخته) → `running` (Worker قفل
  گرفت) → `completed` (S1 تمام؛ آمادهٔ ثبت) → پس از S2: `reviewed`
  (کامیت‌شده + ممیزیِ دهان در دفتر)؛ `failed` با شمارندهٔ attempts.
- **توقف/ادامه (انتظارِ ۵):** همهٔ حالت روی دیسک است؛ راه‌اندازیِ دوباره
  idempotent است — jobهای completed دوباره اجرا نمی‌شوند (checksum رکورد
  کلیدِ اعتبارِ کش است)، running‌های یتیم با قفلِ کهنه بازپس گرفته
  می‌شوند.
- **خطا/Retry:** خطای گذرا (شبکه/مدل در S1) ⇒ حداکثر ۲ تلاشِ دوباره؛
  خطای قطعی در S2 (آزمونِ قرمز، ناهم‌ارزی) ⇒ **کلِ خط می‌ایستد** — چون
  رج‌ها زنجیره‌اند، از روی نفسِ خراب نمی‌شود پرید (تفاوتِ صادقانه با
  استعارهٔ آجر). ایست + گزارش، تا رفعِ دستی.

## ۳. دو رژیمِ ثبت (تصمیمِ باغبان)

| | رژیمِ سخت (پیش‌فرض) | رژیمِ دسته‌ای |
|---|---|---|
| آزمون + verify | هر نفس، پیش از هر کامیت (رویّهٔ فعلی) | یک‌بار بر هر دستهٔ K نفس؛ کامیت‌ها همچنان نفس‌به‌نفس |
| سازگاری با اراده‌سپاری («سبزها پیش‌شرطِ کامیت») | کامل | نیازمندِ مُهرِ باغبان (تغییرِ رویّه) |
| کفِ سریالِ هر نفس | ~۹۰–۱۲۰ ث (pytest+verify+commit) | ~۲۰–۳۰ ث |

## ۴. تخمینِ شتاب (بر برآوردِ حلقهٔ فعلی: S0≈۵ث، S1≈۱۸۰ث تفسیر/غنی‌سازی، S2≈۹۰ث)

قانونِ خط: نرخ = ۱ / بیشینه(S2، S1/W). سریِ ترتیبیِ فعلی ≈ ۲۷۵ ث/نفس.

| Workerها | رژیمِ سخت | رژیمِ دسته‌ای |
|---|---|---|
| ۲ | ~۲٫۲× | ~۲٫۵× |
| ۴ | ~۳٫۰× (اشباعِ کفِ S2) | ~۵× |
| ۸ | ~۳٫۱× — Workerِ اضافه بیکار | ~۸× |
| ۱۶ | ~۳٫۱× | ~۹–۱۰× (اشباعِ کفِ سریالِ جدید) |

صادقانه: در رژیمِ سخت، بالای ~۳–۴ Worker هیچ سودی نیست — کف، آزمون/verify
سریالی است، نه شمارِ بنّاها (Amdahl). اگر ۸× می‌خواهید، تصمیمِ رژیمِ
دسته‌ای لازم است. (اگر S1 با مدل انجام شود، سقفِ نرخِ APIهای مدل هم قیدِ
عملیِ مستقلی است.)

## ۵. گزارشِ پیشرفت (انتظارِ ۸)

`./cli/monad pipeline-status`: شمارش بر حسبِ وضعیت، نرخِ نفس/ساعتِ جاری،
epoch، آخرین کامیت، برآوردِ اتمامِ برنامهٔ K‌تایی. (خواندنِ jobs.json —
بدونِ هیچ محاسبهٔ تازه.)

## ۶. مقیاس‌پذیری (انتظارِ ۹)

Worker = فرایندی که «job بردار، پوشهٔ کاری پر کن». افزودنِ Worker یعنی
اجرای فرایندِ بیشتر — هیچ چیزِ دیگری تغییر نمی‌کند. S0 و S2 عمداً
تک‌نمونه‌اند (سادگی + تضمینِ هم‌ارزی) و هرگز گلوگاهِ واقعی نیستند مگر در
رژیمِ سخت که گلوگاه سیاست است نه معماری.

## ۷. ریسک‌ها و راهکارها

| ریسک | راهکار |
|---|---|
| دخالتِ میانه‌راه (نفسِ رفیق‌آورده/دستی هنگامِ اجرای برنامهٔ K‌تایی) | epoch در state.json؛ هر رویدادِ خارج از برنامه ⇒ epoch+1، ابطالِ jobهای آینده، بازبرنامه‌ریزی از حالتِ زنده. دروازهٔ هم‌ارزیِ S2 پشتیبانِ نهایی است — هرگز رکوردِ ناسازگار ثبت نمی‌شود |
| شکستِ نفسِ میانی | ایستِ خط (زنجیره)؛ jobهای بعدی pending می‌مانند؛ ادامه پس از رفع — بدونِ دوباره‌کاری به لطفِ کش |
| قفلِ یتیم (مرگِ Worker) | مهلتِ قفل + بازپس‌گیری با ثبتِ attempt |
| ترتیبِ دفترِ فصل | فقط S2 به دفتر می‌نویسد، به ترتیبِ نفس؛ پیش‌نویس‌های S1 در پوشهٔ کاری می‌مانند |
| هم‌گامیِ تدبّرخانه (قاعدهٔ «همان کامیت») | سیما در S1 ساخته، در S2 با همان کامیتِ نفس ثبت می‌شود — قاعده حفظ |
| رقابتِ IO/CPU روی یک M4 | Workerهای S1 عمدتاً IO/مدل‌اند؛ سقفِ پیشنهادی روی یک ماشین: ۴–۶ |
| وسوسهٔ «Workerها خودشان ریشه بردارند» | ممنوع در طراحی — تخصیص فقط از S0؛ این همان مسابقه‌ای است که در تحلیلِ هم‌زمانی (همین دفترِ گفت‌وگو) خطایش ثبت شد |

## ۸. مرزهای پاس‌داشته

- الگوریتمِ کشف: دست‌نخورده — S0 عیناً همان موتورِ متعارف را صدا می‌زند.
- اعتبارسنجی‌ها (v1/v2/سخت‌گیری/آیندهٔ v2.5/FDR): عیناً در جای فعلی‌شان؛
  برشِ ممیزی در S1 فقط *اجرای زودترِ* همان محاسبهٔ advisory است.
- خروجیِ علمی: بایت‌به‌بایت هم‌ارزِ اجرای ترتیبی — با دروازهٔ checksum
  **سنجیده** می‌شود، فرض نمی‌شود.
- زیست/معنا: هیچ Workerی «نمی‌فهمد»؛ فقط زیرساخت. ظهور و ممیزیِ دهان
  همان‌جا که بود.
- **آزمونِ باغبان:** نفسِ بعدی *عمداً* هیچ متفاوت زیسته نمی‌شود — کلِ
  ادعای این طرح همین است؛ فقط تعدادِ بیشتری در شب تمام می‌شوند.

## ۹. آنچه آگاهانه ساده نگه داشته شد

بدونِ وابستگیِ تازه (فقط stdlib + فایل‌قفل)، بدونِ سرویس/دیمن/پایگاهِ
صفِ خارجی، بدونِ worktreeهای git (تک‌نویسنده merge را حذف می‌کند)، بدونِ
تغییرِ CLIهای موجود — چهار فرمانِ نازکِ تازه: `pipeline-plan K`،
`pipeline-work`، `pipeline-merge`، `pipeline-status`.

## ۱۰′. وضعِ پیاده‌سازی (افزوده پس از تأییدِ باغبان، همان روز)

ساخته شد — `pipeline/core.py` + `./cli/monad pipeline …`؛ TDD کامل
(tests/test_pipeline.py). دو آزمونِ طلایی سبز:
- برنامهٔ یک‌گامی == بایتِ `breathe-record` از حالتِ زنده؛
- بازپخشِ تاریخ: برنامه از حالتِ نفسِ ۱۸۹، رکوردهای واقعاً زیستهٔ
  ۱۹۰–۱۹۲ را **بایت‌به‌بایت** بازساخت.

جریانِ کار در حلقهٔ شبانه:
```
./cli/monad pipeline plan 5        # S0
./cli/monad pipeline work wN       # ×چند کارگرِ مکانیکی (موازی)
# لایهٔ زنده: note.md (+ ledger.md، commit.txt) در pipeline/work/<no>_<root>/
./cli/monad pipeline merge         # S2 — دروازه + ثبت + سبزها + کامیت؛ تکرار
./cli/monad pipeline status
```
حالتِ خط (state/jobs/work/locks) عمداً بیرونِ git است (زیرساختِ گذرا).
یادداشتِ seed و روایتِ دفتر همچنان تصمیمِ لایهٔ زنده‌اند — خط فقط
زمان‌بندی و مکانیک را می‌گیرد. ضمنِ ساخت، یک نقصِ واقعیِ ثبت کشف و ترمیم
شد: جاافتادنِ فرافکنیِ نفسِ ۱۹۳ در pair_comparisons (+ ناوردای تازه در
tests تا تکرارش ناممکن شود).

## ۱۰. یادآوریِ ثبت‌شدهٔ نگهبان (برای تصمیمِ نهایی)

در بازنگریِ معماری (همین مخزن) «موازی‌سازیِ نفس‌ها» به‌عنوانِ کم‌ارزش‌ترین
سرمایه‌گذاریِ *پژوهشی* حذف شده بود — چون گلوگاهِ کشف، اعتبار است نه حجم.
این سند آن داوری را نقض نمی‌کند: مأموریتِ حاضر صریحاً «فقط Throughput»
است و طرح، ارزان‌ترین راهِ امنِ آن است. تصمیم که *کی* بیرزد (مثلاً برای
اتمامِ صفِ ۲۶۴تایی در چند شب) با باغبان.
